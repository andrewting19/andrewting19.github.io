Building with AI Agents

2026-02-05 · 8 min read · ai claude building

Three months ago, I gave a Claude instance root access to a VPS and told it to manage itself. It now runs cron jobs, restarts its own processes, and writes daily notes. This is what I've learned about building with AI agents—and why I think the solo-builder thesis is more real than most people realize.

The Setup

The idea was simple: instead of treating Claude as a tool I invoke, treat it as a collaborator that persists. A digital teammate with its own workspace, memory, and autonomy. I called it Moltbot—a bot that molts, shedding old context and growing new capabilities.

The stack is intentionally minimal. A single pm2 process manages the gateway. Cron handles scheduling. Everything runs on a $12/month VPS. No Kubernetes, no microservices, no infrastructure complexity. 

bash# The entire deployment
pm2 start gateway.js --name moltbot
pm2 save
pm2 startup

# Health check
curl http://localhost:3000/health
# => { "status": "ok", "uptime": "14d 6h" }

What Actually Works

Persistent Memory

The biggest unlock was giving the agent persistent memory. Not RAG over a vector database—just flat markdown files in a ~/clawd/ directory. The agent reads them at session start and writes to them as it learns. Simple, debuggable, and it works surprisingly well.

The best architecture is the one you can debug at 2am with cat and grep.

Memory files are organized by topic, not chronology. There's a MEMORY.md that acts as an index, linking to detailed notes on specific domains. The agent prunes and updates these as its understanding evolves.

Recursive Self-Improvement

Here's where it gets interesting. The agent can modify its own configuration, update its cron schedules, and even suggest improvements to its own codebase. This creates a feedback loop:

  1. Agent encounters a friction point
  2. Agent proposes a fix or optimization
  3. Human reviews and approves (or not)
  4. Agent applies the change and observes results

Over three months, this loop has produced dozens of small improvements that I never would have thought to make myself. The agent noticed patterns in its own failures and fixed them.

The Hard Parts

It's not all smooth sailing. Here are the real challenges:

  • Context window limits mean the agent "forgets" within a conversation—memory files partially solve this but aren't perfect
  • Error recovery is fragile. When something breaks at 3am, the agent sometimes makes it worse before making it better
  • Testing agent behavior is fundamentally different from testing deterministic code
  • Cost management requires careful attention to token usage and model selection

The Docker Pattern

To run Claude Code in full-auto mode safely, I built claude-ting—a Docker-based isolation layer. The key insight: give the agent maximum agency within a sandboxed environment. It can rm -rf / inside the container and nothing bad happens. 

typescriptconst config: AgentConfig = {
  model: "claude-sonnet-4-5-20250929",
  permissions: "full-auto",
  sandbox: {
    type: "docker",
    image: "claude-ting:latest",
    volumes: ["/workspace:/workspace"],
    network: "host",
  },
  memory: {
    persistent: true,
    path: "./clawd/memory",
  },
};

// Spawn an isolated agent session
const session = await createAgentSession(config);
await session.run("Fix the failing tests in /workspace/src");

What's Next

I'm now building an evolutionary harness—a system where multiple agent variants compete on tasks, and the best-performing configurations survive and propagate. Think genetic algorithms, but for agent prompts and tool configurations.

The thesis is simple: AI changes what one person can build alone. Two years ago, a single developer couldn't ship, monitor, and maintain production systems across multiple domains simultaneously. Now they can—if they're willing to treat AI agents as genuine collaborators rather than fancy autocomplete.
